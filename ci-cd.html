CI/CDとテストの自動化

🚀 1. CI/CDとは何か？

CI/CDは、ソフトウェアの変更を常にテストし、自動的に本番環境へ適用する仕組みのことです。

CI: 継続的インテグレーション

コードが変更されるたびに、自動でビルドとテストを実行します。バグを早期に発見し、「常に動く状態」を保ちます。

目的：統合時のトラブルを防ぐ

CD: 継続的デリバリー / デプロイ

テストを通過したコードを、自動的にテスト環境や本番環境へリリースできるように準備、または適用します。

目的：リリース作業を自動化する

CI/CDパイプラインの全体像

開発からデプロイまでの流れが自動化され、フィードバックループが回る様子

📦 2. バージョン管理 (Git) が出発点

CI/CDの全てのプロセスは、開発者がコードをバージョン管理システム（主にGit）に「プッシュ」することから始まります。

1. コード変更

開発者が手元の環境で機能を実装し、テストコードを書きます。

2. コミット & プッシュ

変更をGitに記録（コミット）し、リモートリポジトリ（GitHubなど）に送信（プッシュ）します。

🚀 トリガー発動！

この「プッシュ」を検知して、CI/CDツールが自動的に動き出します。

⚙️ 3. 具体的なパイプラインの流れ

代表的なCI/CDツールである「GitHub Actions」を例に、具体的な処理の流れを見てみましょう。

プッシュをトリガーに、複数のテストが自動実行されるワークフローの例

ワークフロー

  1. Checkout Code: GitHubから最新のソースコードを取得します。
  2. Run Unit Tests (Jest): Jestなどを使って単体テストを実行し、ロジックの正しさを確認します。
  3. Run Lint Check (ESLint): コードの書き方が規約に合っているか、静的解析を行います。
  4. Run E2E Tests (Playwright): ブラウザを自動操作して、ユーザー視点での動作確認を行います。
  5. Report Status: 全てのテスト結果をプルリクエストの画面などに通知します。失敗すればマージを阻止できます。

✅ 4. 自動化されるテストの種類

CIパイプラインの中では、これまで学習した様々なテスト手法が自動化されて実行されます。

① 単体テスト (Unit)

最も高速に実行される。コミットのたびに毎回走らせるのが基本。

Jest Pytest
② 統合/E2Eテスト

実行に時間がかかるため、重要なブランチへのマージ前などに絞って実行することが多い。

Playwright Cypress
③ 静的解析 (Lint)

コードを実行せずに、バグの温床になる書き方や規約違反をチェックする。

ESLint Prettier
